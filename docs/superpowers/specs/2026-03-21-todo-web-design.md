# TODO Web 设计规格

## 概述

本项目是一个面向作品集展示的全栈 TODO Web 应用，目标是在视觉表现、产品完整度和工程实现之间取得平衡。产品分为营销落地页与登录后应用两部分，既展示产品设计思路，也提供真实可用的任务管理体验。

项目定位为“演示级真实产品”：具备真实前后端、真实数据库、真实认证流程，但避免企业级过度设计，优先保证作品展示效果、可讲述性和开发效率。

## 目标

- 构建一个具备作品集展示价值的全栈 TODO 产品
- 提供真实的用户注册、登录和个人任务管理能力
- 呈现卡片感较强、视觉层次清晰的现代 UI
- 在代码结构上保持清晰分层，便于后续扩展和面试讲解

## 非目标

- 首版不做多人协作
- 首版不做拖拽式看板
- 首版不做实时同步
- 首版不做邮件通知、复杂权限系统或企业级后台能力

## 技术方案

采用以下主技术栈：

- `Next.js App Router`：提供前后端一体化能力
- `PostgreSQL`：作为主数据库
- `Prisma`：负责数据模型和数据库访问
- `Auth.js`：处理登录、会话与路由保护
- `Tailwind CSS + shadcn/ui`：构建现代卡片式界面
- `Zod`：负责表单与输入校验

该方案在作品展示效果、全栈完整度与开发效率之间最均衡，适合作为个人作品集项目。

## 信息架构

### 1. 营销落地页

用于展示产品定位和亮点，建议包含以下区域：

- Hero 区：产品标题、价值主张、主 CTA
- 功能亮点区：任务管理、优先级、到期时间、状态流转
- 产品预览区：展示应用界面 mockup 或截图
- 项目说明区：强调全栈实现、认证、数据库、响应式设计
- 底部 CTA：引导进入注册或登录

### 2. 认证页

- 登录页
- 注册页

风格延续落地页视觉语言，但交互更聚焦转化。首版只做邮箱 + 密码登录注册，不引入第三方登录。

### 3. 应用区 Dashboard

登录后进入主应用，包含：

- 顶部导航：品牌标识、用户信息、退出登录
- 概览卡片：总任务数、已完成、进行中、逾期数
- 筛选区域：全部、待办、进行中、已完成、高优先级
- 任务主列表：展示任务卡片
- 快速新增入口：用于创建任务

## 路由与目录建议

建议采用以下组织方式：

- `app/(marketing)`：落地页相关页面
- `app/(auth)`：登录、注册页面
- `app/app`：登录后的应用页面
- `components/ui`：通用 UI 组件
- `features/todos`：TODO 业务组件、schema 和逻辑
- `lib/auth`：认证配置与会话工具
- `lib/db`：Prisma 客户端与数据库访问公共封装
- `lib/utils`：工具函数

该组织方式能够清晰分离营销展示、认证流程与业务应用，便于后续维护和扩展。

## 数据模型

### User

- `id`
- `name`
- `email`
- `passwordHash`
- `createdAt`
- `updatedAt`

### Todo

- `id`
- `title`
- `description`
- `status`：`todo` / `in_progress` / `done`
- `priority`：`low` / `medium` / `high`
- `dueDate`
- `userId`
- `createdAt`
- `updatedAt`

关系为：`User` 与 `Todo` 一对多。

首版不增加 `Tag`、`Category`、`ActivityLog` 等额外表，以保持模型清晰；但后续可平滑扩展。

## 核心功能范围

### 首版包含

- 用户注册
- 用户登录 / 退出登录
- 仅访问当前用户自己的任务数据
- 创建任务
- 编辑任务
- 删除任务
- 更新任务状态
- 设置任务优先级
- 设置截止日期
- 基础统计卡片
- 状态与优先级筛选
- 响应式布局

### 首版不包含

- 多人共享任务
- 任务评论与附件
- 拖拽排序 / 看板拖拽
- 实时协作
- 邮件提醒

## 关键用户流程

### 首次访问流程

1. 用户进入落地页
2. 点击 CTA 进入注册页
3. 完成注册后自动登录
4. 跳转到应用 Dashboard

### 登录后主流程

1. 系统展示用户自己的任务总览
2. 用户查看统计卡片与任务列表
3. 用户通过筛选器筛选不同任务状态
4. 用户可快速创建、编辑、完成或删除任务

### 创建任务流程

1. 用户点击新增任务按钮
2. 填写标题、描述、优先级、截止日期
3. 提交后进行校验
4. 成功后任务列表与统计同步刷新

### 更新任务流程

- 支持快速切换状态
- 支持打开统一表单修改标题、描述、优先级、截止日期
- 完成任务时提供清晰视觉反馈，例如勾选态、卡片状态标识变化

### 删除任务流程

- 支持删除按钮
- 提供轻量确认，避免误删

## 交互与视觉设计原则

- 以卡片式布局为核心，增强作品展示感
- 保持视觉层次明确：概览、筛选、列表区分清晰
- 仪表盘首屏要让用户快速感知“这是一个完整产品”
- 表单体验统一，减少创建与编辑时的认知切换
- 空状态、错误状态和加载状态都需要具备完整视觉设计

## 工程分层

### UI 层

负责页面展示、交互和用户反馈。包含页面组件、表单组件、统计卡片、任务列表等。

### 业务层

负责 TODO 的创建、更新、删除、筛选与状态流转逻辑。业务规则不应分散在页面组件中。

### 数据访问层

通过 `Prisma` 统一处理数据库读写，避免在页面或零散逻辑中直接操作数据库。

### 认证层

通过 `Auth.js` 管理用户会话、登录状态与路由保护。所有应用区页面均要求登录访问。

## 安全与权限边界

- 所有任务查询必须按当前登录用户的 `userId` 限制
- 所有任务更新与删除操作必须校验归属权
- 未登录用户访问应用区时自动跳转登录页
- 服务端负责最终权限校验，不依赖前端隐藏按钮实现安全控制

## 错误处理策略

- 登录失败：提示邮箱或密码错误
- 注册失败：提示邮箱已存在或字段不合法
- 表单校验失败：就地展示字段错误信息
- 创建 / 编辑失败：通过 toast 或表单提示反馈
- 未授权访问：自动跳转认证页
- 无数据时：展示设计完善的空状态页面
- 服务异常时：展示友好兜底信息，避免暴露底层错误

## 测试策略

首版测试策略保持克制，重点覆盖关键价值路径：

- 单元测试：校验 schema、工具函数、状态映射逻辑
- 集成测试：认证后的任务创建与读取、用户隔离逻辑
- 页面 / 流程验证：登录、注册、Dashboard 主流程

不要求为所有 UI 组件编写测试，也不在首版引入过重的测试体系。

## 里程碑建议

1. 初始化 `Next.js` 项目与基础 UI 框架
2. 配置数据库、Prisma 和 Auth.js
3. 完成落地页
4. 完成认证页
5. 完成 TODO 数据模型与 CRUD
6. 完成 Dashboard 统计、筛选与响应式布局
7. 完成基础测试与收尾优化

## 成功标准

- 用户能够完成注册、登录和退出
- 用户只能看到和操作自己的任务
- TODO CRUD、状态更新、优先级与截止日期工作正常
- 落地页与应用区视觉统一且具作品展示感
- 代码结构清晰，能清楚说明认证、数据库与业务分层方案

## 结论

本项目将实现为一个带营销落地页的全栈 TODO 产品。它应具备真实的认证与数据库能力，并通过卡片式视觉和完整的 Dashboard 体验展现产品成熟度。首版以任务管理核心能力为主，聚焦在“作品可展示、功能可运行、架构可讲述”这三个目标上。
