# TODO Web 国际化与主题系统设计规格

## 概述

本规格定义 TODO Web 下一阶段的产品与工程演进，目标是在现有全栈 TODO 应用基础上引入完整的中英文国际化能力、全站主题切换能力，以及围绕这两项能力展开的高价值 UI 调整。

本阶段聚焦三件事：

- 建立规范的 `zh` / `en` 国际化体系
- 建立 `light` / `dark` / `system` 的主题切换体系
- 将语言切换、主题切换与全站视觉统一整合进高质量 UI

本次改动不是局部补丁，而是把国际化与主题切换提升为全站平台能力，以支撑后续继续扩展页面、功能和视觉细节。

## 目标

- 支持 `zh` 和 `en` 两种语言
- 使用 locale 路由结构表达当前语言状态
- 页面文案、服务端用户可见提示文案均支持国际化
- 支持 `light` / `dark` / `system` 三种主题模式
- 主题切换支持持久化，并默认跟随系统
- 通过统一 Header 暴露语言与主题切换入口
- 统一 Landing、Auth、Dashboard 的主题适配与视觉风格
- 保持代码边界清晰，使 i18n 与主题系统易于后续扩展

## 非目标

- 本阶段不引入更多语言
- 不做复杂的日期、数字、时区本地化深度定制
- 不做多语言 SEO 深度优化
- 不做大规模品牌重设计或整站重画
- 不做与本阶段目标无关的布局重构

## 技术方案

采用以下主方案：

- `next-intl`：负责 App Router 下的国际化
- `next-themes`：负责主题切换与持久化
- `middleware.ts`：统一处理 locale 识别、语言重定向和受保护路由逻辑
- `messages/zh.json` 与 `messages/en.json`：集中存放翻译文案
- 根级 Provider：统一为服务端与客户端组件提供 locale 与 theme 能力

该方案能够兼顾结构清晰度、展示效果与未来可扩展性，适合当前项目阶段。

## 路由与语言策略

### Locale 路由

应用改为以 `/{locale}` 为根的路由结构，仅支持：

- `zh`
- `en`

建议结构：

- `app/[locale]/(marketing)`
- `app/[locale]/(auth)`
- `app/[locale]/app`

### 根路径策略

- 用户访问 `/` 时，不直接渲染页面内容
- 由 `middleware` 或入口重定向逻辑导向对应语言路径

### 语言来源优先级

建议按以下优先级决定当前语言：

1. 用户手动选择后持久化的 locale cookie
2. 浏览器语言
3. 默认语言 `en`

### 语言切换行为

- 顶部 Header 提供语言下拉菜单
- 用户切换语言后：
  - 更新当前 URL 到对应 `/{locale}` 路径
  - 更新 locale cookie
  - 页面文案、导航、错误提示同步切换

## 主题策略

### 主题模式

支持以下三种模式：

- `light`
- `dark`
- `system`

### 默认行为

- 默认使用 `system`
- 用户手动切换后，优先使用用户选择并持久化

### 主题切换入口

- Header 提供主题切换入口
- 主题切换与语言切换并列放置
- 切换应即时生效，并能在刷新后保持状态

### 视觉实现原则

- 页面不直接依赖硬编码深色样式
- 改为围绕统一主题 token、CSS 变量或等价样式策略构建
- 所有核心页面在 light / dark 模式下都必须保持良好可读性与层次感

## 文案组织策略

文案统一存放于：

- `messages/zh.json`
- `messages/en.json`

建议按领域组织 key：

- `common`
- `marketing`
- `auth`
- `dashboard`
- `todos`
- `errors`
- `theme`
- `locale`

这样有助于后续维护、查找和扩展，也方便不同页面只依赖自己关注的文案域。

## 服务端提示文案策略

本阶段要求服务端用户可见提示也支持国际化，因此建议：

- 服务端逻辑不直接写死中文或英文句子
- 服务端 action / route 返回稳定错误码或错误类型
- 页面层或响应适配层根据当前 locale 将错误码映射为翻译文案

建议覆盖的用户可见场景包括：

- 登录失败
- 注册失败
- 表单校验失败
- 未授权访问
- Todo 不存在或不可操作
- 通用服务异常

这种“错误码 + 翻译映射”的策略能让业务逻辑与具体文案解耦，减少未来扩语言或改文案的维护成本。

## UI 调整范围

### 1. 全站 Header

Header 升级为全站统一导航入口，负责承载：

- 品牌标识
- 语言切换下拉菜单
- 主题切换菜单
- 登录入口 / 用户态入口

Header 要在 `light` 与 `dark` 下都具有一致而清晰的视觉层级。

### 2. Landing Page

本阶段对 Landing Page 的调整重点是：

- 所有文案进入国际化体系
- 背景、卡片、按钮、文字对比度适配深浅主题
- 保持 Hero、功能区、产品预览区的视觉统一性

### 3. Auth 页面

登录页与注册页需要：

- 全文案国际化
- 错误提示国际化
- 卡片、输入框、按钮在 light / dark 模式下都保持质感和可读性
- 与 Landing 视觉语言保持统一

### 4. Dashboard 与 Todo 相关 UI

本阶段需覆盖：

- Dashboard 标题与辅助文案
- 统计卡片
- 筛选区域
- Todo 卡片
- 新建任务表单
- 空状态与常见交互提示

重点是去掉当前偏向“仅深色优先”的视觉假设，使其在 light 模式下也自然成立。

## 组件与模块边界

### i18n 层

负责：

- locale 配置
- 消息加载
- 语言路由解析
- 与 App Router 的集成

不应直接承载业务逻辑。

### Theme 层

负责：

- 主题 Provider
- 当前主题状态
- 持久化
- `system` / `light` / `dark` 选择逻辑

页面组件只消费主题能力，不自行实现主题判定。

### Navigation / Shell 层

负责：

- 全站 Header
- 语言切换控件
- 主题切换控件
- 认证入口或用户态展示

### Feature 文案消费层

- `marketing`
- `auth`
- `todos`

各功能域只应读取自己需要的翻译 key，避免一个组件知道整站所有文案结构。

### 服务端 Action / Route 层

负责：

- 接收输入
- 执行业务
- 返回成功结果、稳定错误码和必要数据

不直接承担具体自然语言文案输出。

## 数据流

### 页面访问

1. 用户访问站点
2. `middleware` 判断 locale
3. 请求导向 `/{locale}` 对应路由
4. 页面通过国际化 Provider 获取文案
5. 页面通过主题 Provider 获取主题能力

### 语言切换

1. 用户在 Header 中选择语言
2. 客户端触发 locale 路由切换
3. 更新 locale cookie
4. 页面重新渲染为对应语言内容

### 主题切换

1. 用户在 Header 中选择主题模式
2. `next-themes` 更新主题状态
3. 将主题选择持久化
4. 页面即时更新视觉样式

### 业务操作与错误提示

1. 用户提交登录、注册或 TODO 表单
2. 服务端执行 action / route
3. 若失败，返回稳定错误码
4. 页面根据当前 locale 映射出最终提示文案

## 错误处理策略

### 输入校验错误

- 在 schema 层定义
- 页面根据翻译 key 展示字段级错误

### 业务错误

例如：

- 邮箱已存在
- 邮箱或密码错误
- 未授权访问
- Todo 不存在

这些错误通过服务端稳定错误码返回，再映射为翻译文案。

### 系统兜底错误

- 向用户展示通用且友好的失败提示
- 不暴露底层技术错误信息

## 测试策略

### 单元测试

覆盖：

- locale 配置与消息加载
- 错误码到翻译 key 的映射
- 主题和语言切换相关工具函数

### 集成测试

覆盖：

- 注册 / 登录 / Todo 操作返回稳定错误码
- 在不同 locale 下映射出正确提示文案

### E2E / Smoke 测试

至少验证以下关键路径：

- 默认访问时进入正确 locale
- 切换语言后 URL 与文案同步变化
- 切换主题后页面样式状态变化并持久化
- 未登录访问受保护路由时，在对应语言下进入正确登录页
- 至少验证 `zh` 与 `en` 各一条主路径

## 实施顺序

建议按以下顺序实施：

1. 搭建 `next-intl` 基础能力
2. 搭建 `next-themes` 基础能力
3. 升级全站 Header
4. 改造 Landing 与 Auth 页面
5. 改造 Dashboard 与 Todo 相关 UI
6. 收敛服务端错误为“错误码 + 翻译映射”结构
7. 完成回归测试与视觉收尾

## 成功标准

- 用户可在 `zh` 与 `en` 之间切换
- URL 能表达当前语言状态
- 页面文案和用户可见服务端提示都随语言切换
- 用户可在 `light` / `dark` / `system` 之间切换
- 主题选择刷新后仍保留
- Header 成为统一的语言与主题切换入口
- Landing、Auth、Dashboard 在 light / dark 下均具有良好可读性
- 现有代码结构在 i18n 和主题能力接入后仍保持清晰边界

## 结论

本阶段将 TODO Web 从“单语言、深色优先”的作品展示型应用，升级为一个具备完整国际化能力与全站主题能力的更成熟产品。通过 `next-intl`、`next-themes`、统一 Header、消息文件和稳定错误码映射策略，本项目将在用户体验、工程结构和作品集展示说服力上显著提升。
